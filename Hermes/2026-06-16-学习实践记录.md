## 一、Hermes 安装问题修复

### 问题
Hermes 更新时 `npm ci` 在 desktop 阶段失败，Electron 下载报 `TypeError: fetch failed`

### 原因
- npm 未配置代理（用户网络需要走 `127.0.0.1:7892`）
- Electron 的 postinstall 脚本独立下载二进制，不认 npm 的 proxy 设置
- 需要设环境变量 `HTTPS_PROXY` + `ELECTRON_GET_USE_PROXY=1`

### 解决方案
```bash
# 1. 配置 npm 代理
npm config set proxy http://127.0.0.1:7892
npm config set https-proxy http://127.0.0.1:7892

# 2. 设置环境变量让 Electron 下载走代理
export HTTPS_PROXY=http://127.0.0.1:7892
export HTTP_PROXY=http://127.0.0.1:7892
export ELECTRON_GET_USE_PROXY=1

# 3. 重新安装依赖
cd ~/AppData/Local/hermes/hermes-agent
npm ci

# 4. 如果 electron 二进制被删了，单独重新安装
node node_modules/electron/install.js
```

### Playwright Chromium 问题
- 国内网络无法下载 Playwright 的 Chrome for Testing（Google Cloud Storage / CDN 均被墙）
- 各种镜像（npmmirror、Azure CDN）要么 404 要么 400
- **不影响 Hermes CLI 核心功能**，浏览器工具可通过 CDP 连接系统 Chrome 替代

## 二、Chrome 远程调试（CDP）

### 工作原理
Chrome DevTools Protocol (CDP) 允许外部程序通过 WebSocket 连接控制 Chrome 浏览器。

### 启动方式

**方式 A：Hermes 自带函数（稳定，一秒连接）**
```python
from hermes_cli.browser_connect import try_launch_chrome_debug
try_launch_chrome_debug(port=9222)
# 会使用临时配置文件 ~/.hermes/chrome-debug/
```

**方式 B：桌面快捷方式加参数（方便长期使用）**
```
"C:\Program Files\Google\Chrome\Application\chrome.exe" --remote-debugging-port=9222 --remote-allow-origins=* --remote-debugging-address=127.0.0.1
```

### Chrome 151 Bug
- `--remote-debugging-port=9222` 参数被接受但端口实际未打开（HTTP 502）
- 需要额外加 `--remote-allow-origins=*` 和 `--remote-debugging-address=127.0.0.1`
- 即使加了也可能出现连接不稳定、断线后无法重连的问题
- `try_launch_chrome_debug` 方式始终稳定工作

### 浏览器自动化工具
Hermes 提供以下 browser 工具：
- `browser_navigate(url)` — 导航到页面
- `browser_snapshot()` — 获取无障碍树快照
- `browser_click(ref)` — 点击元素（ref 可能动态变化，配合 snapshot 使用）
- `browser_console(expression=js)` — 执行 JS 并获取结果（最稳定的交互方式）
- `browser_type(ref, text)` — 输入文本

### 实际验证
- **携程订票**：CDP 方式访问 flights.ctrip.com，可以正常加载页面和执行 JS
  - ❌ 携程有反爬机制，搜索结果 API 被屏蔽（"未找到符合条件的航班"）
  - ✅ 用 Hermes 自带启动方式（临时配置）则正常
- **IPSA 报销系统**：成功爬取 66 条已通过报销记录，汇总金额 ¥13,193.00

## 三、桌面宠物（桌宠）原理

### 核心原理
桌宠 = **透明窗口 + 序列帧动画 + 行为逻辑**

### 技术分层
1. **透明窗口** — `SetWindowLong` + `WS_EX_LAYERED`，或 Electron 的 `transparent: true`
2. **鼠标穿透** — `WS_EX_TRANSPARENT`，点击透明区域 = 点到桌面
3. **序列帧动画** — 多张图片轮流播放（和 GIF 一个原理）
4. **行为逻辑** — 状态机/行为树决定下一步动作

### 实现方式对比

| 方式 | 优点 | 缺点 |
|------|------|------|
| **Electron** | 前端技术栈，动画丰富，好扩展 | 包大 (~150MB) |
| **Python tkinter** | 轻量，无需额外安装 | 效果简陋，透明度支持有限 |
| **PyQt5** | 性能好，透明度完美 | 需安装 PyQt5 |
| **Live2D** | 效果最逼真（抖音上那种） | 需要专业模型（付费） |
| **Canvas 绘制** | 灵活可控 | 画出来的始终是图画，非真实 |

### 比熊桌宠 Demo
- 尝试了 SVG 矢量画 → 太卡通，用户不满意
- 升级为 Canvas 分层渲染（渐变、毛绒质感、骨骼动画、眼睛跟随鼠标）
- 最终结论：要真正像抖音那种效果，需要 Live2D 模型 + 专业美术

### 桌宠行为状态机
```
站立(idle) ↔ 走路(walk) ↔ 开心(happy)
   ↕            ↕
  好奇(curious) ↔ 睡觉(sleepy)
   ↕
  兴奋(excited)
```
- 自动状态切换（定时 + 概率）
- 鼠标交互（点击、拖拽、右键）
- 眼睛跟随鼠标移动
